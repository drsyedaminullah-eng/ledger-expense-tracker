## 2. AI Code Audit & Attribution

**AI Tool Used:** Claude (Anthropic) — claude.ai

**The "Veto":**
During development, the AI's initial approach to rendering expense descriptions used `innerHTML` to insert user-provided text directly into the DOM. For example:

```javascript
// ❌ INSECURE — AI's original suggestion
expenseItem.innerHTML = `<div class="expense-name">${userDescription}</div>`;
```

This pattern is a classic **Cross-Site Scripting (XSS)** vector. If a user entered `<script>alert('hacked')</script>` as a description, it would execute in the browser. I corrected this in two ways:

1. All input is first sanitized through **DOMPurify** before being stored.
2. When rendering stored descriptions back to the page, I use **`textContent`** (not `innerHTML`) so the browser never parses the value as HTML — even if something slipped past sanitization.

```javascript
// ✅ SECURE — corrected approach
const el = document.getElementById('ename-' + e.id);
if (el) el.textContent = e.desc; // textContent never executes scripts
```

## 3. Hardening Measures (The "Big Two")

| Feature | Implementation Detail | Threat Mitigated |
|---|---|---|
| **Input Sanitization** | All user-provided text (expense descriptions) is passed through the **DOMPurify** library before storage, and rendered back to the DOM via `textContent` instead of `innerHTML`. A category whitelist check also prevents unexpected values from being stored. | Cross-Site Scripting (XSS) — OWASP A03:2021 |
| **Password Hashing + Salt** | Passwords are never stored in plain text. On registration, a random 16-byte **salt** is generated using `crypto.getRandomValues()`. The password is then hashed as `SHA-256(salt + password + salt)` using the browser's native **Web Crypto API** (`crypto.subtle.digest`). Only the hash and salt are saved — never the raw password. | Credential Exposure / Insecure Password Storage — OWASP A02:2021 |

### Additional Defensive Measures

- **Input length limits:** All text fields have `maxlength` attributes enforced at the HTML level (descriptions capped at 60 chars, passwords at 128 chars) to prevent buffer-overflow style abuse and denial-of-service via massive inputs.
- **Amount validation:** The expense amount is parsed as a float and validated to be a positive number between 0.01 and 999,999 — rejecting strings, negative values, and non-numeric input server-side (in JS logic), not just in the UI.
- **Session vs. Persistent Storage:** The logged-in session is stored in `sessionStorage` (expires when the tab closes), not `localStorage`. Only hashed credentials and expense data (no raw passwords) use `localStorage`.
- **Username allowlist regex:** Usernames are validated against `/^[a-zA-Z0-9_]{3,32}$/` — only alphanumeric characters and underscores are accepted, preventing injection characters in usernames.

## 4. Dependency Disclosure

| Library | Version | Purpose |
|---|---|---|
| DOMPurify | 3.0.6 | XSS sanitization of user-generated text |
| Web Crypto API | Built-in (browser) | SHA-256 password hashing and salt generation |
| Google Fonts | CDN | Typography (Syne, DM Mono) — no user data processed |

**Vulnerability Check:** DOMPurify 3.0.6 was verified against known CVEs at the time of development. The Web Crypto API is a browser built-in with no external dependency risk. Google Fonts CDN loads display assets only.

> To verify DOMPurify yourself: visit [https://security.snyk.io/package/npm/dompurify](https://security.snyk.io/package/npm/dompurify) and confirm no critical vulnerabilities exist for version 3.0.6.
